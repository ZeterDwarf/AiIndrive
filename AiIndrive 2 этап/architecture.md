# Архитектура решения: AgroScoring AI

## 1. Введение
Данный документ описывает высокоуровневую архитектуру решения для скоринга сельхозпроизводителей. Проект реализуется в рамках хакатона Decentrathon 5.0 (Кейс 2).
Главная цель: переход от принципа «кто первый встал» к системе распределения субсидий на основе объективных данных об эффективности хозяйства (Merit-based подходу).

## 2. Компоненты системы

Система состоит из следующих логических модулей:

### 2.1. Data Ingestion Layer (Сбор и предобработка данных)
- **Функция:** Принимает сырые данные из базы данных или CSV-файлов, выданных заказчиком.
- **Действия:** Очистка от пропусков, выбросов. Нормализация данных, приведение к нужному формату (числовые и категориальные признаки).

### 2.2. Feature Engineering Module (Генерация признаков)
- **Функция:** Создание новых, значимых для бизнеса признаков (отражающих суть задачи субсидирования).
- **Примеры признаков:** 
  - `yield_trend` — стабильность роста урожайности.
  - `subsidy_efficiency` — отношение полученных ранее субсидий к росту налоговых отчислений или выпуска продукции.
  - `risk_factor` — наличие налоговых долгов или штрафов.

### 2.3. ML Scoring Engine (Ядро скоринга)
- **Функция:** Предсказание вероятности эффективного использования запрашиваемых субсидий.
- **Алгоритмы:** Для текущего MVP используется **RandomForestClassifier** (Ансамбль деревьев решений). В дальнейшем модель будет переведена на CatBoost/LightGBM для улучшения качества и встроенной поддержки категориальных фичей.
- **Выход:** Итоговый Score от 0 до 100 для каждого заявителя.

### 2.4. Explainability Module (Модуль объяснимости)
- **Критически важное требование ТЗ** (ИИ не должен быть черным ящиком).
- **Рационализация:** Использование деревьев решений позволяет сразу же применять подходы на основе Feature Importances и SHAP (SHapley Additive exPlanations) для генерации текстовых причин ("Почему данному кандидату выставлен балл 85?").
- **Выход:** Список понятных текстовых аргументов для комиссии (например: "Отсутствие задолженностей и стабильный рост урожайности > 5% за последние годы").

### 2.5. API / Integration Layer (в планах на 2-ой этап)
- **Функция:** FastAPI микросервис, который будет принимать запросы с параметрами заявителя (JSON) и возвращать результаты скоринга с объяснениями.

## 3. Workflow (Поток данных для оценки)
1. Заявитель (или комиссия) отправляет данные заявки в систему.
2. Данные проходят через пайплайн предобработки.
3. Модель (RandomForest / CatBoost) оценивает кандидата.
4. Модуль Explainability вычисляет причины (локальное объяснение).
5. Система формирует итоговый JSON / отчет: Score + Decision + Explainability.
6. Человек-эксперт принимает финальное решение (AI как ассистент).
